# 贡献指南 (Contributing)

感谢您对 **VISTA-Grasp** 的关注与喜爱！我们深知一个开源工具的繁荣离不开社区的力量，因此我们非常欢迎来自开源社区的 Bug 报告、功能建议以及代码贡献。

为了保障项目的代码质量与合规性，提升沟通效率，请在参与前仔细阅读以下指南。

## 1. 贡献前准备

- 请先查阅本项目的 `README.md` 以及位于 `docs/` 目录下的进阶架构文档，确保对项目的模块划分有基础的了解。
- 在您准备大干一场之前，请前往 [GitHub Issues](#) 检索一下，确认您要解决的问题或开发的新功能没有被其他人认领或已经处于修复状态中。

## 2. 贡献流程

### 2.1 提交 Issue
无论是发现 Bug 还是构思了绝妙的新功能，都请通过提交 Issue 来与我们沟通。
- **Bug 报告**：请务必包含：清晰的复现步骤、报错日志上下文、您的系统环境信息（如 Ubuntu 版本、VR 设备型号等）。
- **功能需求 (Feature Request)**：请描述该功能解决的痛点场景、预期行为，以及任何有助于实现该功能的参考链接或设计思路。

### 2.2 提交 Pull Request (PR)
我们采用标准的开源 Fork 开发工作流：

1. Fork 本项目的远程仓库到您的个人 GitHub 下。
2. 在您的 Fork 仓库中，拉取目标主干分支的最新的代码（通常是 `main` 或 `teleop-recode`）。
3. 切出一个具有描述性名字的特性分支：`git checkout -b feature/awesome-new-hand-tracking` 或 `fix/webrtc-black-screen`。
4. 编写您的代码，并进行本地测试。强烈建议在 Dummy 模式跑通之后，能在可能的条件下使用真实硬件或仿真进行联调测试。
5. 提交代码变更。请遵循常规的 Angular Commit Message 规范，例如 `feat: xxx`，`fix: xxx`，`docs: xxx`。
6. 将分支推送到您的远程仓库，并向我们的官方主仓库发起 Pull Request。
7. 耐心等待维护人员进行 Code Review。

## 3. 代码审查与安全合规要求

作为企业级开源项目，我们对 PR 的合规性有以下红线要求：

- **代码规范**：
  - Backend (Python) 代码请遵循 PEP-8 规范。
  - Frontend (React/TS) 代码请确保没有 ESLint 报错，并尽量使用强类型定义。
- **⚠️ 敏感信息清理**：在提交任何配置文件、脚本或代码之前，请绝对确保其中没有遗留内部的私有 IP 地址、服务器测试账号、密码、API 密钥等任何公司敏感信息！
- **贡献者许可协议 (CLA)**：
  - 在合并您的 PR 之前，所有外部贡献者均需要签署我们的 [贡献者许可协议 (CLA)](CLA.md)。
  - 如果只是修复错别字或几十行的微小修改，您可以简单地在 Git 提交记录中使用 `Signed-off-by` 来替代繁琐的签署流程。使用 `git commit -s` 即可自动添加。

## 4. 获取支持与联系我们

如果您在二次开发或环境配置中遇到难以解决的阻碍，可以通过发 Issue 联系我们，或者直接发送邮件至开源技术支持邮箱：`yucongcong654@gmail.com`。
