# 系统配置与自定义端口指南

本项目的所有核心配置项均通过项目根目录下的 `.env` 文件进行统一管理。在启动系统前，建议您根据实际的网络与物理环境进行核对或调整。

## 环境配置文件

项目根目录下提供了两个配置模板供参考：
- `.env.example`：完整配置项模板（包含追踪模式、超时配置、调试等级等所有可用参数）。
- `.env.example.lite`：精简版模板（仅包含必填的网络与核心端口配置，适合快速验证）。

**配置方法**：
```bash
cp .env.example .env
# 使用你喜欢的编辑器修改 .env 文件，修改后重启对应服务即可生效。
```

## 系统端口占用说明

完整的遥操系统默认占用以下三个核心端口，如果您的服务器或开发机存在端口冲突，请在 `.env` 中修改，并确保它们未被防火墙拦截：

| 默认端口 | 运行侧与服务 | 用途说明 |
| --- | --- | --- |
| `5173` | 前端 Vite Server | WebXR 访问入口。戴上 VR 头显后在浏览器中输入的 URL 端口。 |
| `5174` | 开发机后端 `TeleopServer` | 信令枢纽。处理前端传来的 WebSocket 动作指令，并进行 WebRTC (SDP/ICE) 握手协商。 |
| `5175` | 机器人端桥接服务 | 指令下发通道。Python 后端通过该 WebSocket 连接将 IK 解算后的关节指令发送给机器人；在使用真实硬件时，该端口通常通过 SSH Local Forwarding 映射到开发机本地。 |

## 核心配置项解析

以下是 `.env` 中的常见核心参数说明：

- **`TELEOP_MODE`**: 遥操追踪输入模式。
  - `controller_mode` (默认)：使用双手柄（Controllers）进行空间位姿追踪与扳机键（控制夹爪）读取，稳定性最高。
  - `hand_mode`：启用纯手部追踪（Hand Tracking），依赖于头显的手势识别算法，无需握持物理手柄。
- **`TELEOP_ROBOT_PORT`**: 后端用于连接机器人桥接服务的端口，对应上述表格中的 5175。
- **`TELEOP_CONDA_ENV`**: `run/start.sh` 等 Tmux 自动化脚本启动时需要激活的 Conda 虚拟环境名称，默认为 `teleop`。
